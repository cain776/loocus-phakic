의료 데이터에 맞춘 보안 체계

비쥬웍스 아이차트는 의료 데이터의 특수성을 전제로 설계되었습니다. 편의보다 책임 추적 가능성과 무결성을 우선으로 두고 구조화된 보안 체계를 갖추고 있습니다.
🛡

병원이 요구하는 보안은 다릅니다

의료 데이터에 특화된 엄격한 접근 제어, 개인정보보호법·의료법 준수, 내부자 오남용 사후 감사까지 — 비쥬웍스는 "누가, 언제, 어떤 환자 정보를, 어떻게 다뤘는가"가 빠짐없이 기록되는 체계를 목표로 설계하고 있습니다.

[보안 1] 🔐 인증 · 접근 제어 (Authentication & RBAC) 운영 중
"누가 들어왔는가"를 넘어, "그가 무엇을 볼 권리가 있는가"까지 제어합니다.

적용된 보안 체계

  • bcrypt 해시 기반 비밀번호 저장 — 복호화 불가, 평문 노출 원천 차단
  • 환경변수 기반 비밀키 관리 — 소스 코드에 하드코딩 절대 금지
  • 역할(Role) 기반 접근 제어(RBAC) — 역할별 권한 매트릭스로 민감 화면을 명시적 허용 시에만 노출
  • 단일 세션 정책 — 동일 계정 중복 로그인 차단. 신규 로그인 시 이전 세션 자동 로그아웃
  • Login_History 전수 기록 — 모든 로그인 시도를 IP · 기기 · 시각 · 성공/실패까지 남김
  • 비밀번호 변경 시 타 세션 무효화 — 계정 탈취 의심 시 즉시 차단 가능
법적 근거: 개인정보보호법 제29조(안전조치 의무) 및 의료법 제21조에 따른 접근통제 · 접근권한 관리 요건 충족.
[보안 2] 🗝 민감 정보 암호화 · 보호 운영 중
주민번호 등 고유식별정보는 DB 레벨 암호화로 저장되고, 요청 경로는 보안 헤더·CSRF·Rate Limit으로 다층 보호됩니다.

적용된 보안 체계

  • 주민번호 DB 레벨 암호화 저장 — 단순 DB 덤프만으로는 복호화 불가
  • 주요 HTTP 보안 헤더 일괄 적용 — X-Frame-Options · X-Content-Type-Options 등을 helmet 미들웨어로 처리
  • CSRF 토큰 검증 — 세션 탈취를 통한 위조 요청 차단
  • Rate Limiting — 로그인 · 주요 API 엔드포인트에 요청 수 제한으로 무차별 대입 방지
  • 입력값 검증 자동화 — 77개 Zod 스키마가 HTTP 경계에서 SQL Injection · XSS 패턴 차단
  • SQL 파라미터 바인딩 의무화 — 문자열 결합 쿼리 작성 자체를 개발 표준에서 금지

※ 암호화 키 관리 방식(패스프레이즈 분리·KMS 도입 등)은 도입 시 Phase 1에서 현황 재점검 후 강화 방안을 확정합니다.

법적 근거: 개인정보보호법 제24조(고유식별정보 처리 제한) 및 「개인정보 안전성 확보조치 기준」 제7조(개인정보의 암호화)에 따른 암호화 저장 의무 준수.
[보안 3] 📋 감사 로그 (Audit Trail) · 책임 추적성 운영 중
"누가 언제 어떤 요청을 했는가"가 요청 단위로 역추적 가능합니다. 환자 정보 접근 이력은 확장 계획 단계에서 강화합니다.

적용된 보안 체계

  • 모든 요청에 reqId 부여 — 하나의 요청이 어떤 사용자로부터 시작되어 어떤 DB 쿼리까지 도달했는지 역추적 가능
  • requestLogger 미들웨어 — userId · 세션 ID · IP · latency 자동 기록
  • globalErrorHandler 중앙 집계 — 모든 예외가 표준 포맷으로 포착 · 분류
  • 민감 정보 로깅 금지 규칙 — 주민번호 · 비밀번호 등은 로그에 절대 기록되지 않도록 개발 표준에서 명시

확장 계획 Phase 1 설계

  • 환자 정보 조회 이력 테이블 — 누가 언제 어떤 환자를 열었는지 별도 기록. 도입 시 Phase 1에서 스키마·보존기간 확정
  • 중앙 집중 로그 수집 — requestLogger 출력을 중앙 플랫폼으로 통합 보관 · 주기 점검
법적 근거: 「개인정보 안전성 확보조치 기준」 제8조(접속기록의 보관 및 점검) — 접속기록 2년 이상 보관 · 월 1회 이상 점검 의무 대응.
[보안 4] ⚙ 보안 미들웨어 파이프라인 — 자동 적용 운영 중
모든 요청은 라우트 작성자가 신경 쓰지 않아도 7계층 보안 검사를 통과한 뒤에야 비즈니스 로직에 도달합니다.

7계층 보안 항목

  • 1. helmet — HTTP 보안 헤더 자동 부여
  • 2. CORS — 허용된 도메인에서만 API 호출 가능
  • 3. sessionConfig — 세션 유효성 검증
  • 4. limiter — Rate Limit + 로그인 전용 강화 Limit
  • 5. csrf — CSRF 토큰 검증
  • 6. requireAuth — 인증 확인 후에만 라우트 진입
  • 7. globalErrorHandler — 예외 중앙 집계 · 민감 정보 누출 방지

※ 파이프라인에는 이외에도 responseHandler(응답 포맷 통일), requestLogger(감사 로그) 등이 포함되어 있으며, 신규 API를 추가해도 이 7계층 보안은 자동으로 적용됩니다.

[보안 5] 🔄 동시성 · 데이터 무결성 운영 중 + 계획
여러 직원이 같은 환자를 동시에 다뤄도 데이터가 조용히 손실되지 않도록 구조적으로 보호합니다.

운영 중인 보호 장치

  • Save-Lock 안전 패턴 — beginSave/finishSave는 try/finally로 쌍을 맞추는 것을 개발 표준에서 강제하여 UI 영구 잠금 예방
  • SQL 파라미터 바인딩 표준 — 동시 실행 시 데이터 손상·주입 경로 차단
  • Zod 스키마 입력 검증 — 경계에서 불일치 데이터 유입 차단

도입 계획 Phase 1~3 단계 도입

  • Optimistic Lock (버전 필드) 기반 덮어쓰기 방지 — "다른 사용자가 먼저 수정했습니다" 알림 + 병합 선택권
  • 예약 슬롯 복합 유니크 제약 — 시간 + 의사 + 장비 기준으로 이중 배정 원천 차단
  • Idempotency Key — 네트워크 지연에 의한 수납 중복 요청 차단

자세한 시나리오와 대응 로직은 동시성 / 데이터 정합성 페이지 참조.

[보안 6] 📐 코드 레벨 보안 규율 운영 중
보안은 라이브러리 설치로 끝나지 않습니다. 코드가 작성되는 순간부터 규칙이 강제됩니다.

개발 표준에 내재화된 보안 규칙

  • 하드코딩 비밀번호 금지 — 코드 리뷰 체크리스트 필수 항목
  • SQL 문자열 결합 금지 — 파라미터 바인딩만 허용
  • innerHTML 사용 시 escapeHtml 의무화 — XSS 차단
  • 콘솔에 민감 정보 출력 금지 — 주민번호·비밀번호 로깅 원천 차단
  • 사용자 입력 필수 검증 — 모든 경계에서 스키마 검증 통과 의무
  • 보안 변경은 오너 승인 필수 — 비밀번호 정책·인증 테이블 변경은 단독 진행 금지
운영 자산: 이 규칙은 모두 개발 표준 문서(CLAUDE.md)에 명문화되어 있으며, 신규 개발자는 첫 PR부터 동일한 규칙을 따르게 됩니다.

비쥬웍스가 보안에서 자신 있게 말할 수 있는 이유

★ 보안은 기능이 아니라 규율입니다
  • "설치하면 안전한" 보안이 아니라 "개발할 때마다 강제되는" 보안 — 신규 기능을 추가할 때마다 7계층 미들웨어 · 77개 스키마 · 개발 표준이 자동으로 적용됩니다.
  • "로그가 있음"이 아니라 "역추적이 가능함" — reqId 하나로 사용자 요청 → 서비스 호출 → DB 쿼리까지 시간 순으로 복원할 수 있습니다.
  • "보안 담당자에 의존"이 아니라 "규칙에 의존" — CLAUDE.md에 명문화된 규칙은 사람이 바뀌어도 그대로 유지됩니다.
  • "사고 후 추적"이 아니라 "사고 전 차단" — Rate Limit, CSRF 토큰, SQL 파라미터 바인딩, escapeHtml 등은 사건이 발생하기 전에 막습니다.
의료 데이터는 유출되면 되돌릴 수 없습니다.
비쥬웍스는 이 되돌릴 수 없는 위험을 "편의보다 우선"으로 두고 시스템을 설계했으며, 이 결정이 레거시 환경과 아이차트의 가장 큰 차이를 만듭니다.
