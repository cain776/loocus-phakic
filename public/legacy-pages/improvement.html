IMPROVEMENT — VISUWORKS APPROACH

개선 방안

현재 문제 분석에서 진단된 8개 문제를, 기술적 기반에서 확보한 자산으로 해결합니다. 각 해결 방안은 "우리가 이미 검증한 방법"에 근거합니다.

접근 원칙: 과장된 기술 스택 나열이 아닌, ✓ 이미 운영 중인 방식과 ◇ 도입 검토 중인 방식을 구분해 정직하게 제시합니다. 도입 단계는 로드맵을 참조하십시오.

[개선 1] Web 기반 단일 고객뷰 — 언제 어디서든 운영 중
Windows 전용 Fat Client → 브라우저만 있으면 접근 가능한 Web 플랫폼
대응 문제: [문제 1] Windows 전용 Fat Client 구조 — 업무 이동성 전무, 수동 배포

우리의 해결 방식

  • Express + Vanilla JS Web 애플리케이션 — 서버 1회 배포로 모든 단말에 즉시 반영
  • 단일 고객뷰 — 환자 한 명 선택 = 상담·검사·수술·리콜·SMS 이력을 한 화면에서
  • Windows · Mac · 태블릿 · 모바일 브라우저 지원 → 원내/원격/재택 동일 업무 환경
  • Route/Service/Schema 3레이어로 라우트 82개, Service 31개, Zod 스키마 77개가 일관된 규칙으로 동작
기반 자산 (foundation §1): Route / Service / Schema 3레이어 표준화, CLAUDE.md 문서화된 개발 규율
[개선 2] 단일 입력 · 통합 데이터 모델 운영 중
성민CRM + MD소프트 + 종이차트 3중 입력 → 1회 입력으로 전 과정 흐름
대응 문제: [문제 2] 시스템 분산 · 데이터 이중 입력 — 오타·누락·통합 분석 불가

우리의 해결 방식

  • 통합 고객 마스터 — 환자 등록 1회로 예약·검사·상담·수술·리콜 전 과정 연결
  • SOFTCRM · MCRM · IPLUS 멀티 DB 통합 운영 — 단일 커넥션 풀 + 크로스 DB 쿼리로 분산 데이터도 끊김 없이 조회
  • Zod 기반 입력 검증 — 77개 스키마가 HTTP 경계에서 데이터 품질을 보장
  • Customer 도메인은 customer-read/write/response/sms/visit-motive 등 7개 서비스로 책임 분리

데이터 품질 개선 계획 검토 중

  • RTF 기반 CUSTOM_MEMO → 구조화 텍스트 필드 병행 도입 (검색 가능성 확보)
  • 코드 체계 정합화 — 프론트/DB 값 불일치 제거 (파일럿 현황 분석 단계에서 확정)
  • MSSQL → PostgreSQL 점진 이관 검토 — 오픈소스 전환 및 전문 검색 기능 확보
기반 자산 (foundation §1, §2): 도메인별 서비스 분리(Customer 7개), 멀티 DB 통합 운영 경험
[개선 3] 측정 기반 성능 최적화 운영 중
인덱스 전략 부재로 p95 1.75~14초 → 커버링 인덱스 실측 기반 80~97% 단축
대응 문제: [문제 3] 검색·조회 성능 병목 — 상담 중 대기, 리콜 업무 병목

우리의 해결 방식

저희는 "빨라진 것 같다"를 인정하지 않습니다. 모든 성능 개선은 k6 시나리오와 smoke 테스트로 전후 p95 latency를 비교하여 검증합니다.

  • k6 시나리오 03~06 — 고객 검색, 리콜 등 핵심 플로우를 정량 측정
  • Smoke 테스트 5종 — auth / login-history / customer-all-panels / customer-update / prod-compare
  • 커버링 인덱스 전략 — 자주 쓰는 조건과 반환 컬럼을 포함해 테이블 조회 없이 인덱스만으로 응답
  • Slow Query 감지 문화 — 문제가 된 쿼리를 분해·재작성 (예: SP_CUSTOM_MEMO 14초 → 0.9초)

RBAC 세분화 · 감사 로그 · 단일 세션 정책 — 개인정보보호법·의료법 대응 기반
대응 문제: [문제 4] 보안 · 컴플라이언스 취약 — RBAC 부족, 감사 로그 부재, 내부자 위협 탐지 불가

인증 · 접근 제어

  • ECP_Auth 기반 인증 — bcrypt 해시, 환경변수 기반 비밀키 관리 (하드코딩 제거)
  • 단일 세션 정책 — 한 계정의 중복 로그인을 차단해 책임 추적성 확보
  • Login_History 이력 기록 — 로그인 시각·IP·결과를 모두 저장
  • 역할 기반 접근 제어 — 직원별 권한 분리, 민감 화면은 명시적 허용만 통과

데이터 보호

  • 주민번호는 DECRYPTBYPASSPHRASE 기반 암호화 저장 (개인정보보호법 준수)
  • helmet 미들웨어로 보안 헤더 일괄 적용 (X-Frame-Options, X-Content-Type-Options 등)
  • CSRF 토큰 · Rate Limit이 미들웨어 파이프라인에 내장되어 라우트 작성 없이 자동 적용
기반 자산 (foundation §1): helmet → cors → session → limiter → csrf → requireAuth 표준 파이프라인
[개선 5] 표준화된 확장 구조 — 커스텀 개발 최소화 운영 중
신규 기능 = 커스텀 개발 → Route/Service/Schema 표준으로 신속 추가
대응 문제: [문제 5] 통합 · 확장성 부재 — 장비 연동, 신규 서비스마다 유상 커스텀 개발

우리의 해결 방식

  • 표준 구조 기반 신속 개발 — 신규 엔드포인트 추가 시 Route/Service/Schema 3개 파일만 작성, 미들웨어는 자동 적용
  • 일관된 패턴 — 모든 개발자가 동일 규칙을 따르므로 신규 인력 온보딩 비용이 낮음
  • CLAUDE.md 문서화된 표준 — AI 개발 어시스턴트도 동일 규칙으로 코드를 생성

장비 · 외부 연동 계획 단계 도입

  • 검사장비 연동 — 파일럿 단계에서 장비 프로토콜 확정 후 Service 레이어로 표준 연동
  • CTI 콜센터 연동 — 전화 수신 이벤트 → 고객 자동 조회 (MSA 워커 분리 검토)
  • SMS/카카오톡/이메일 채널 확장 — 알림 서비스 모듈로 분리
기반 자산 (foundation §1): Route/Service/Schema 3레이어, 31개 Service 모듈 분리 경험
[개선 6] 요청 단위 추적 가능한 운영 체계 운영 중
장애 원인 파악 지연 → reqId 하나로 요청→서비스→DB 전 구간 역추적
대응 문제: [문제 6] 관측성 · 운영 체계 부재 — 에러는 사용자 제보로만 인지

우리의 해결 방식

  • 구조화된 로깅 — 모든 요청에 reqId, userId, 세션 컨텍스트 자동 태깅
  • requestLogger 미들웨어 + req.logger — 개발자는 로깅 코드 없이도 컨텍스트 로그가 기록됨
  • globalErrorHandler — 모든 예외가 한 곳에서 포착되어 표준 포맷으로 응답 + 기록
  • k6/smoke 정기 실행 — 성능 회귀를 조기 감지

중앙 관측 플랫폼 확장 계획 검토 중

  • 중앙 로그 수집 · 대시보드 시각화 (Loki/Grafana 등 오픈소스 기반)
  • 메트릭 기반 임계 알림 — p95 급등, 에러율 임계 초과 시 자동 알림
  • 장애 이력 DB — 동일 이슈 반복 발생 방지
기반 자산 (foundation §3): 구조화 로깅 + requestLogger + globalErrorHandler 체계

핵심 메시지

★ 왜 이 개선 방안이 신뢰할 수 있는가
  • "가설"이 아닌 "운영 실적" — 3, 4, 6번 개선은 이미 저희 운영 시스템에서 검증된 방식입니다.
  • "과장"이 아닌 "단계" — 아직 검토 중인 항목(PostgreSQL, 중앙 관측, 장비 연동)은 명확히 구분해서 표기했습니다.
  • "도구"가 아닌 "기반" — 기술 스택 나열이 아니라, 청주김안과의 8가지 문제에 정확히 매핑된 해결책입니다.
AI 시대의 시스템 선택 기준은 "무엇을 할 수 있느냐"가 아니라 "그 위에서 무엇을 키울 수 있느냐"입니다.
저희가 제안드리는 시스템은, 청주김안과가 오늘 도입해 내일 AI를 얹을 수 있는 이미 준비된 기반입니다.
