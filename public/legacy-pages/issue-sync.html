DATA INTEGRITY — CONCURRENCY

동시성 / 데이터 정합성

여러 직원이 같은 환자·같은 차트를 동시에 조작할 때 발생하는 레거시 환경의 구조적 위험과 대응 방안

왜 중요한가: 병원은 상담사·데스크·간호사·원장이 같은 환자의 차트를 동시에 열고 수정하는 환경입니다. 레거시 환경에서는 동시성 처리가 "없거나, 마지막 쓰기가 이긴다(Last-Write-Wins)" 수준이라 누가 무엇을 덮어썼는지도 모르는 데이터 손실이 조용히 누적됩니다.

[이슈 1] Last-Write-Wins — 조용한 데이터 덮어쓰기 Critical
두 사용자가 동시에 같은 환자 정보를 수정하면 나중에 저장한 쪽이 앞의 변경을 통째로 덮어씁니다. 경고도, 기록도 남지 않습니다.

발생 시나리오

09:10 상담사 A 환자 김OO 조회 → 메모 수정 중
09:11 데스크 B 동일 환자 조회 → 전화번호 수정 저장
09:12 상담사 A 메모 저장 → B가 수정한 전화번호가 사라짐
09:15 저녁에 B가 발견, 원인 알 수 없음 → CS 이슈

구조적 원인

  • 낙관적 잠금(Optimistic Lock) 또는 버전 필드 부재
  • 저장 시 "내가 조회한 시점과 지금 DB가 같은가?" 검증 없음
  • 전체 레코드 UPDATE — 변경한 필드만 반영하는 Partial Update 부재
비쥬웍스 대응 방식

✓ 버전 필드(`updated_at` 또는 `version`)를 모든 고객 테이블에 도입하여, 저장 시점에 원본과 다르면 "다른 사용자가 먼저 수정했습니다"를 알리고 병합 선택권을 제공합니다. Service 레이어에서 일관되게 검증하여 라우트마다 재구현할 필요가 없습니다.

[이슈 2] 예약 이중 배정 — 같은 시간, 두 환자 Critical
예약 등록 시 시간/의사/장비 충돌 검증이 없거나, 검증과 저장 사이의 Race Condition으로 같은 슬롯에 두 환자가 배정됩니다.

발생 시나리오

10:00 데스크 A 14:00 OCT 검사 예약 등록 중
10:00 데스크 B 14:00 OCT 검사 예약 등록 중 (다른 창구)
10:00 A, B 동시에 "저장" 클릭 → 둘 다 저장 성공
14:00 두 환자가 같이 와서 현장에서 갈등

구조적 원인

  • 예약 슬롯에 유니크 제약 없음 (시간 + 의사 + 장비)
  • "빈 슬롯 조회 → 사용자 확인 → 저장" 사이의 시간이 길어 Race 발생
  • DB 트랜잭션 격리 수준(Isolation Level) 부적절
비쥬웍스 대응 방식

✓ 예약 슬롯 테이블에 복합 유니크 제약 (datetime + doctor + device)을 걸어 DB 레벨에서 이중 배정을 원천 차단합니다. 저장 실패 시 "이미 배정되었습니다"를 즉시 사용자에게 안내하여 현장 갈등을 방지합니다.

[이슈 3] 수납 금액 중복 처리 High
네트워크 지연으로 사용자가 "수납" 버튼을 여러 번 누르면 같은 건이 2~3번 저장되어 금액이 곱절로 기록됩니다.

발생 시나리오

14:30 데스크 수납 버튼 클릭 (네트워크 느림)
14:31 반응 없음 → 다시 클릭
14:32 세 번째 클릭
14:33 서버 응답 도착 → 같은 수납 3건 생성
월말 결산에서 발견 → 환불·사과 필요

구조적 원인

  • 멱등성(Idempotency) 보장 부재 — 같은 요청의 중복 실행 구분 불가
  • 버튼 이중 클릭 방지 UI 패턴 없음
  • 서버 측 중복 차단 로직 부재
비쥬웍스 대응 방식

✓ 수납·저장 요청에 Idempotency Key를 부여하고 서버가 동일 키를 기억했다가 중복 요청을 차단합니다. UI는 저장 중 버튼을 비활성화하여 사용자 실수를 예방합니다.

[이슈 4] 세션 공유 / 다중 로그인 High
한 계정으로 여러 PC에서 동시에 로그인하면, 누가 어떤 작업을 했는지 추적할 수 없어 의료정보 접근 책임이 불명확해집니다.

발생 시나리오

  • 직원 ID를 팀원들이 공유 사용
  • A가 자리 비운 사이 B가 A 계정으로 조회
  • 감사 로그에 "A가 조회"만 남음 — 실제는 B
  • 정보 유출 의심 시 조사 불가
비쥬웍스 대응 방식

✓ 단일 세션 정책 — 신규 로그인 시 이전 세션을 자동 로그아웃. ✓ Login_History에 IP·기기·로그인 시각을 모두 기록. → 책임 추적 가능성과 단독 사용 원칙을 동시에 확보합니다.

[이슈 5] 크로스 DB JOIN 일관성 — SOFTCRM ↔ MCRM ↔ IPLUS High
Linked Server로 여러 DB를 결합해 조회할 때, 각 DB의 최신성이 달라 화면마다 다른 값이 보이는 경우가 있습니다.

구조적 원인

  • DB 간 동기화 지연 (수 초 ~ 분 단위)
  • 트랜잭션 경계가 단일 DB에 한정 — 분산 트랜잭션 부재
  • 조회 캐시가 화면·모듈마다 다른 시점의 데이터 보관
비쥬웍스 대응 방식

✓ 단일 고객뷰를 제공하는 Service 레이어가 DB별 조회 시점을 통일 관리. ✓ 중요 업무(수납·수술 확정)는 명시적 트랜잭션 범위에서만 진행. ✓ 장기적으로는 이벤트 기반 동기화로 전환 검토.

[이슈 6] Save-Lock 영구 잠금 — UI가 멈춘 듯이 보임 Medium
저장 중 예외가 발생하면 "저장 중" 상태가 해제되지 않아 다시 저장할 수 없게 됩니다. 사용자는 새로고침을 강제당합니다.

구조적 원인

  • beginSave() 호출 후 예외 발생 시 finishSave()가 호출되지 않음
  • try/finally 패턴 누락
비쥬웍스 대응 방식

✓ CustomerStore 캡슐화 규칙으로 beginSave는 반드시 try/finally와 쌍을 이루도록 강제합니다. ✓ 자동화된 캡슐화 점검 도구로 회귀를 차단합니다. (비쥬웍스가 내부 운영 규칙으로 이미 적용 중)
