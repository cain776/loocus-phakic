상담사·데스크·간호사·원장이 같은 환자 차트를 동시에 다뤄도 — 누가 무엇을 덮어썼는지 모르는 데이터 손실이 발생하지 않도록, 코드 표준·DB 제약·예외 처리·롤백 전략이 다층으로 맞물려 동작합니다.

여러 직원이 같은 환자의 정보를 동시에 수정해도 한쪽 변경이 조용히 사라지지 않도록 구조화합니다.
전형적 사고 시나리오 09:10 상담사 A가 환자 김○○ 메모 수정 중 · 09:11 데스크 B가 같은 환자 전화번호 수정 저장 · 09:12 A가 메모 저장 → B의 전화번호 수정이 통째로 사라짐. 경고도, 기록도 없음.

운영 중인 보호 장치

  • Save-Lock 안전 패턴 — beginSave/finishSave는 try/finally로 쌍을 맞추는 것을 개발 표준이 강제하여 UI 영구 잠금 예방
  • CustomerStore 캡슐화 규칙 — 외부에서 내부 상태 직접 수정을 차단하여 동시 수정 시 예측 불가 상태 방지
  • 자동화된 캡슐화 점검 도구 — 코드 변경 시 회귀를 자동 감지
  • SQL 파라미터 바인딩 의무화 — 동시 실행 중 쿼리 꼬임 · 주입 경로 차단

확장 계획 Phase 1~3 단계 도입

  • Optimistic Lock (버전 필드) — 저장 시 원본과 다르면 "다른 사용자가 먼저 수정했습니다" 알림 + 병합 선택권
  • 예약 슬롯 복합 유니크 제약 — 시간 + 의사 + 장비 기준 이중 배정 원천 차단
  • Idempotency Key — 네트워크 지연으로 수납 버튼을 여러 번 눌러도 중복 처리되지 않음

※ 자세한 시나리오는 동시성 / 데이터 정합성 페이지 참조.

[안정성 2] ⚠ 예외 · 장애 중앙 처리 운영 중
오류가 발생해도 정보 누출 없이 표준 포맷으로 포착되어, 담당자가 즉시 원인을 파악할 수 있습니다.

운영 중인 구조

  • globalErrorHandler — 모든 예외가 한 곳에서 포착 · 분류되어 라우트마다 try/catch를 반복 작성할 필요 없음
  • 표준 에러 응답 포맷 — 내부 스택 트레이스 · 경로 · DB 에러 메시지가 클라이언트로 그대로 노출되지 않음
  • 민감 정보 로깅 금지 규칙 — 주민번호 · 비밀번호가 에러 로그에 포함되지 않도록 개발 표준에서 명시
  • 요청 단위 reqId 역추적 — 에러 발생 시 사용자 요청 → 서비스 호출 → DB 쿼리 전 구간을 시간 순으로 복원
실무 효과 "오후 3시쯤 수납 화면에서 에러 났다고 하는데…" — 담당자가 reqId 하나로 해당 시간대·사용자·요청·쿼리를 즉시 조회. "어디서부터 문제인지 모른다"가 없는 구조입니다.
[안정성 3] 📊 성능 · 기능 회귀 감지 운영 중
"예전보다 느려진 것 같아요"를 감(感)이 아닌 정량 지표로 잡아냅니다.

운영 중인 검증 자산

  • k6 부하 시나리오 — 고객 검색 · 리콜 조회 등 핵심 플로우의 p50/p95/p99 정기 측정
  • Smoke 테스트 5종 — auth · login-history · customer-all-panels · customer-update · prod-compare
  • WORK ↔ PROD 응답 비교 — 배포 전후 핵심 응답이 동일한지 자동 비교로 회귀 조기 발견
  • 실측 개선 이력 — 커버링 인덱스 도입으로 p95 최대 -97% (3.3초 → 0.1초) 개선을 수치로 증빙

청주김안과 도입 시 효과

  • 초기 이관 후 p95 지표를 기준선(baseline)으로 확보
  • 이후 기능 추가마다 k6로 회귀 여부를 검증 — "어느 날 갑자기 느려졌다"를 예방
  • 원장님이 체감하는 속도 변화를 수치로 설명할 수 있음
[안정성 4] ↩ 안전한 전환 · 롤백 전략 도입 단계
새 시스템으로의 전환은 "되돌릴 수 없는 결정"이 아닙니다. 문제가 생기면 즉시 이전 방식으로 복귀할 수 있도록 설계합니다.

도입 로드맵에 반영된 안전 장치

  • 레거시 시스템 읽기 전용 병행 보존 — 전면 오픈 후에도 기존 성민CRM을 즉시 폐기하지 않고 조회 가능 상태로 유지
  • D-day 전일 전수 백업 — 전환 직전 DB · 파일 · 설정 전체 스냅샷 확보
  • 이관 후 교차 검증 리포트 — 건수 · 합계 · 샘플 일치율을 서면으로 확인한 뒤에만 다음 단계 진행
  • 파일럿 Go/No-Go 기준 — Critical 버그 0건 · 성능 목표 충족 · 만족도 통과를 모두 만족해야 전면 오픈
  • W12 원내 상주 지원 — 전환 직후 한 주간 비쥬웍스 엔지니어가 원내에 상주하며 즉시 대응

※ 상세 전환 단계는 도입 단계별 로드맵 참조.

[안정성 5] 📐 안정성을 코드로 강제하는 개발 규율 운영 중
안정성은 "신경 쓰면 지켜지는" 것이 아니라 "규칙이 자동으로 강제하는" 것이어야 합니다.

개발 표준에 내재화된 규칙

  • Route → Service → Schema 3레이어 강제 — 모든 신규 기능이 일관된 구조로 작성되어, 어디서 문제가 나도 찾기 쉬움
  • 77개 Zod 입력 검증 스키마 — 잘못된 데이터가 DB에 닿기 전에 경계에서 차단
  • SQL 문자열 결합 금지 — 파라미터 바인딩만 허용. 주입 경로뿐 아니라 데이터 꼬임도 예방
  • save-lock / 캡슐화 회귀 점검 도구 — 규칙 위반 시 자동 감지
  • 보안 · 인증 변경은 오너 승인 필수 — 핵심 자산의 단독 변경 차단
신규 개발자 온보딩 예시 개발 표준 문서를 따라 첫 PR을 작성하면 — try/catch 없이도 에러가 중앙 처리되고, 검증 스키마가 자동 적용되고, 로깅이 자동 붙습니다. "개발자가 잊어도 안정성이 누락되지 않는 구조"입니다.

레거시 vs 아이차트 — 안정성 수준 비교

안정성 항목 레거시 (C# + MSSQL) 아이차트 (EyeChartPro)
동시 수정 충돌 Last-Write-Wins (조용한 덮어쓰기) Save-Lock + Optimistic Lock(계획)
예약 이중 배정 Race Condition으로 발생 가능 복합 유니크 제약 도입 계획
수납 중복 처리 버튼 반복 클릭 시 다중 저장 Idempotency Key 도입 계획
예외 처리 화면마다 개별 try/catch · 누락 빈번 globalErrorHandler 중앙 집계
에러 원인 추적 사용자 제보 → 개발자 수동 로그 탐색 reqId 하나로 전 구간 시간 순 복원
회귀 감지 "느려진 것 같다" 체감으로만 인지 k6 + smoke 정기 정량 측정
배포 롤백 수동 복구 · 전환 시 이전 시스템 폐기 병행 운영 + Go/No-Go 기준 + 전수 백업
입력값 안전성 화면마다 수동 검증 · 누락 가능 77개 스키마 경계 일괄 검증

비쥬웍스가 안정성을 말할 수 있는 이유

★ 안정성은 기능이 아니라 구조입니다
  • "실수가 없어야 안정"이 아니라 "실수해도 데이터가 남는" 구조 — Save-Lock, Optimistic Lock(계획), 복합 유니크 제약(계획)이 사람의 주의력에 의존하지 않습니다.
  • "장애가 없어야 안정"이 아니라 "장애를 빠르게 추적 가능한" 구조 — reqId 하나로 모든 구간을 역추적할 수 있으니 "원인을 모른다"가 사라집니다.
  • "느려지지 않아야 안정"이 아니라 "느려졌음을 수치로 감지하는" 구조 — k6/smoke 체계가 감(感)이 아닌 기준선(baseline)으로 판단합니다.
  • "전환은 되돌릴 수 없다"가 아니라 "문제 생기면 바로 복귀 가능한" 구조 — 레거시 병행 보존과 Go/No-Go 기준이 도입 리스크 자체를 낮춥니다.
병원에서 가장 두려운 것은 "눈에 보이는 장애"가 아니라 "눈에 보이지 않는 데이터 손실"입니다.
비쥬웍스는 이 조용한 손실을 막기 위해 — 기능보다 규율, 편의보다 무결성을 우선합니다.
